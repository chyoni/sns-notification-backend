# 알림 서버를 세 개의 모듈 (api, consumer, core)로 나눈 이유

- 우선, MSA 환경에서 댓글 서버, 리액션 서버, 팔로우 서버가 있다고 가정한 상태에서 알림 서버를 구현
- 사용자가 자신의 알림을 확인하기 위해 알림 화면을 호출하는 행위가 발생
- 사용자가 자신의 알림을 확인하기 위해 알림 조회 API를 처리하는 것과, 각 서버에서 이벤트를 받아 이벤트를 처리하는 행위를
하나의 서버가 부담하면 여러 문제가 발생할 수 있다.
- 예측 가능한 문제는, 조회 요청이 갑자기 많이 들어올 경우 조회 요청을 처리하다 서버가 버티지 못하면 이벤트를 받아 데이터를
저장하는 쓰기 작업에 문제를 줄 수 있고 이는 곧 하나의 서버가 너무 많은 부담을 지게 되는 구조
- 그렇기에 조회 및 API 처리 서버와 이벤트를 컨슘하여 데이터 생성하는 쓰기 서버를 분리한다.
- 이 두 서버는 모두 같은 Core를 사용하기에(엔티티, 데이터베이스, 비즈니스 핵심 로직) 코어 모듈을 분리하여 재사용 가능하도록 구현한다.

# 외부 서비스와의 통신을 REST API가 아닌 이벤트 기반의 아키텍처로 구성한 이유 (한마디로 카프카를 사용하는 이유)
- 이벤트를 생성하는 각 서버(댓글, 리액션, 팔로우 서버)는 이제 이벤트를 소모하는 consumer 서버와만 직접적인 관계가 있다.
- 그렇지만, 여전히 각 서버끼리 강결합의 문제가 있다. 만약 댓글이 너무 많이 달리는 초대형 인플루언서라고 생각해보면,
1초에도 댓글이 수십만개가 달릴수도 있다. 이때 댓글서버와 Consumer 서버가 서로 동기적으로 REST API를 통해 주고 받는다면
Consumer 서버가 처리하는 처리량에 따라 댓글 서버에 영향을 주고, 그 반대로도 마찬가지가 된다. 설령 이 처리량을 감당하지 못해
Consumer 서버가 죽는다면 댓글 서버뿐 아니라 리액션, 팔로우 서버도 문제가 발생한다. 즉, 단일 실패 지점이 명확하게 존재한다.
- 동기적으로 수행되는 작업이 아니라고 해도 마찬가지다. 댓글 서버가 REST API로 요청만 날리고 응답을 기다리지 않는다고 해도,
그 수많은 요청을 감당하지 못한 Consumer 서버는 서버가 죽게되면 다른 서버에도 역시나 영향이 간다. 
- 이러한 문제들을 극복하고자, 서버간에 강결합을 느슨한 결합(Loose Coupling)으로 전환하기 위해 이벤트 기반의 아키텍처를 구현한다.
- 이를 위해 카프카를 도입한다. 이제 댓글 서버든, 리액션 서버든 본인의 비즈니스 로직에만 집중할 수 있고 이벤트를 생성하여 카프카에게 전송만하면
그 이후는 더이상 관심사가 아니다. 이제 Consumer 서버는 이벤트를 지속적으로 가져와 이벤트를 소모하고 데이터를 생성한다.
- 여기서 메시지 큐(예를 들면 RabbitMQ)는 왜 사용하지 않는가?에 대한 질문을 한다면 내 대답은 다음과 같다.
메시지큐와 카프카의 가장 근본적인 차이는 카프카는 Append-Only라는 사실이다. 즉, 이벤트를 쓰기만 하고 삭제하지 않는다. 따라서, 
갑자기 너무 많은 요청이 몰려 Consumer 서버가 죽더라도, 본인이 읽은 이벤트부터 다시 읽어가면 된다. 또는 이 경우 이벤트의 유실은
없었는지 판단하기에도 카프카가 적합하다. 메시지큐는 결국, 한번 소모하면 해당 메시지를 삭제하기 때문에 이러한 내구성을 가지고 있지 않다.

# RDB와 MongoDB 둘 중 어느것을 사용하나?

- 기본으로 RDB를 가져갈 생각이다. 다만, 이 알림 서버 프로젝트가 클린 아키텍처로 구성된 점을 고려해
의도적으로 두 개의 전혀 성격이 다른 데이터 관리 시스템을 모두 사용해, 얼마나 기술 교체에 유연한지를 몸소 느껴볼 예정이다.

# 통합테스트에 테스트 컨테이너를 사용한 이유

- 최대한 동일한 환경의 테스트 환경을 만드는 것이 최우선 원칙이었다. 간단하게 H2 데이터베이스를 사용해도 큰 무리는 없을것으로
예상되지만, 위에서도 말했듯 난 지금 MongoDB와 MySQL 이 두가지 서로 다른 스토리지 기술을 사용해 볼 예정이기에 MongoDB의
그때 그때 유연하고 빠르게 테스트 코드를 작성할 수 있는 테스트 컨테이너를 활용하기로 했다.

# ADR-005: 클린 아키텍처 / 헥사고날 아키텍처 적용

| 항목 | 내용 |
|------|------|
| 상태 | 승인됨 (Accepted) |
| 날짜 | 2026-04-09 |
| 의사결정자 | cwchoiit |

## 컨텍스트 (Context)

포트폴리오 프로젝트로서 두 가지 설계 목표가 있다.

1. **테스트 용이성**: 프레임워크나 DB 없이 비즈니스 규칙을 단독으로 테스트할 수 있어야 한다.
2. **기술 교체 유연성**: ADR-003에서 결정한 듀얼 스토리지 전략처럼, 인프라 기술을 교체해도 도메인 로직에 영향이 없어야 한다.

전통적인 Layered Architecture(Controller → Service → Repository)에서는 Service가 JPA Repository에 직접 의존하고, 비즈니스 로직이 프레임워크 어노테이션에 오염된다. 이 구조에서는 DB 없이 Service 로직을 테스트하거나, 스토리지 기술을 교체하기 어렵다.

## 결정 (Decision)

Robert C. Martin의 **클린 아키텍처**와 Alistair Cockburn의 **헥사고날 아키텍처(Ports and Adapters)**를 결합하여 적용한다.

**의존성 규칙 (Dependency Rule):**
```
[Infrastructure / Adapter] → [Application] → [Domain]
```
모든 의존성은 안쪽(Domain 방향)으로만 흐른다. 바깥 레이어가 안쪽 레이어를 알 수 있지만, 그 반대는 허용되지 않는다.

**레이어 구조와 역할:**

| 레이어 | 위치 | 역할 |
|--------|------|------|
| Domain | `core/.../domain/` | 순수 비즈니스 엔티티와 규칙. 프레임워크 의존 없음 |
| Application (Port In) | `core/.../application/port/in/` | UseCase 인터페이스 — 외부에서 애플리케이션에 진입하는 계약 |
| Application (Port Out) | `core/.../application/port/out/` | Repository 인터페이스 — 애플리케이션이 인프라에 요청하는 계약 |
| Application (Service) | `core/.../application/notification/` | UseCase 구현체. Port In을 구현하고, Port Out에만 의존 |
| Adapter In | `consumer/.../adpater/in/` | Kafka Consumer. 외부 이벤트를 받아 UseCase 호출 |
| Adapter Out | `core/.../adapter/out/`, `consumer/.../adpater/out/` | Port Out 구현체. DB/외부 API 호출 |
| Infrastructure | `core/.../infrastructure/` | Spring 설정, DB 엔티티 등 순수 기술 관심사 |

**모듈 간 Input Adapter 분리 (의도적 설계):**

`notification-core`는 Input Adapter를 포함하지 않는다. Input Adapter(REST Controller, Kafka Consumer)는 각각 `notification-api`와 `notification-consumer`에 위치한다. `notification-core`는 Port(UseCase 인터페이스)만 제공하며, 어떤 방식으로 진입하든 동일한 UseCase를 호출한다.

**실용적 트레이드오프:**

Application Service에 `@Service` 어노테이션을 사용한다. 순수한 클린 아키텍처에서는 Application 레이어가 프레임워크에 의존하지 않아야 하지만, Spring DI 컨테이너에 빈으로 등록하기 위한 최소한의 의존으로 허용한다. 이는 `@Configuration` 클래스에서 수동 등록하는 방식의 보일러플레이트보다 실용적이다.

## 대안 (Alternatives Considered)

**전통적 Layered Architecture:**
- Controller → Service → Repository JPA의 전통적인 3계층 구조.
- 간단하지만 Service가 JPA에 직접 의존하여 테스트와 기술 교체가 어렵다.
- 채택하지 않음.

**DDD (Domain-Driven Design with Aggregates):**
- Aggregate 단위로 도메인을 모델링하는 보다 정교한 접근.
- 현재 프로젝트 규모와 복잡도에 비해 과도한 설계다.
- 채택하지 않음.

## 결과 (Consequences)

### 긍정적 결과
- **Domain 레이어 완전 격리**: `Notification`, `CommentNotification` 등 도메인 클래스는 프레임워크 의존성이 전혀 없어 순수 Java로 테스트 가능하다.
- **Output Port를 통한 인프라 교체**: ADR-003에서 MongoDB → MySQL 교체 실증의 기반이 된다.
- **테스트 더블 용이**: `NotificationInMemoryRepositoryAdapter`가 `NotificationRepository`를 구현하므로 단위 테스트에서 DB 없이 Service 로직을 검증할 수 있다.

### 부정적 결과 / 트레이드오프
- Port 인터페이스, 어댑터 구현체, 엔티티 매핑 코드 등 보일러플레이트 코드가 증가한다.
- Application Service의 `@Service` 어노테이션으로 인해 완전한 프레임워크 격리는 달성하지 못했다.

### 알려진 이슈

| # | 내용 | 심각도 | 권장 조치 |
|---|------|--------|-----------|
| 1 | `notification-consumer`의 패키지명 오타: `adpater` (→ `adapter`여야 함) | 중간 | 리팩토링으로 수정 |
| 2 | `NotificationMongoRepository` (Spring Data 인터페이스)가 `adapter.out.persistence`에 위치 | 낮음 | `infrastructure.mongo.persistence`로 이동 고려 |
| 3 | Application Service에 `@Service` 사용 | 낮음 | 실용적 트레이드오프로 허용 |

### 후속 과제
- `adpater` 패키지명 오타 수정
- LIKE, FOLLOW 이벤트에 대한 어댑터 매핑 완성 (현재 `toDomain()`/`toEntity()`에 `// TODO: LIKE, FOLLOW` 주석 존재)
- `notification-api`에 알림 조회 REST Controller 구현

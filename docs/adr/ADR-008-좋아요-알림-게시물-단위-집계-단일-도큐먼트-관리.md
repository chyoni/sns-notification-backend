# ADR-008: 좋아요 알림을 게시물 단위로 집계하여 단일 도큐먼트로 관리한다

| 항목 | 내용 |
|------|------|
| 상태 | 승인됨 (Accepted) |
| 날짜 | 2026-04-14 |
| 의사결정자 | cwchoiit |

## 컨텍스트 (Context)

기존 구조에서는 좋아요 이벤트가 발생할 때마다 `LikeNotification` 도큐먼트를 1개씩 생성했다. 동일 게시물에 N명이 좋아요를 누르면 N개의 도큐먼트가 쌓이는 구조다. 알림 수신자 입장에서 "A님이 좋아요를 눌렀습니다", "B님이 좋아요를 눌렀습니다"가 별개 알림으로 계속 추가되는 것은 자연스러운 UX가 아니다. 일반적인 SNS 알림은 "A님, B님 외 3명이 좋아요를 눌렀습니다"처럼 같은 게시물에 대한 좋아요를 하나로 집계해서 보여준다. 이 집계 표현을 구현하려면 도메인 모델과 이벤트 처리 방식을 함께 바꿔야 한다.

## 결정 (Decision)

게시물 하나당 `LikeNotification` 도큐먼트를 1개만 유지하고, 좋아요를 누른 유저 ID를 `likedIdsBy: List<Long>`에 임베드하는 집계 구조를 채택한다.

**도메인 변경:**
- `likedBy: Long` → `likedIdsBy: List<Long>` — 여러 유저 ID를 하나의 도큐먼트에 보관
- `addLikedId(userId, occurredAt)` — 중복 없이 ID를 추가하고 `occurredAt`·`expiresAt`을 최신 시각으로 갱신
- `removeLikedId(userId)` — 좋아요 취소 시 ID 제거
- `likedCount()` — 현재 좋아요 수 반환

**이벤트 처리 변경 (`LikeEventService`):**
- 좋아요 발생 시: `findLikeByPostId`로 기존 도큐먼트를 조회한다. 있으면 `addLikedId` 후 저장(upsert), 없으면 새 도큐먼트를 생성한다.
- 좋아요 취소 시: `findLikeByPostId`로 도큐먼트를 조회하고 `removeLikedId`를 호출한다. `likedCount()`가 0이 되면 도큐먼트를 삭제한다.

**조회 추가:**
- `findLikeByPostId(postId)` — 게시물 ID만으로 집계 알림을 조회하는 메서드를 포트·어댑터 전 레이어에 추가한다.

## 대안 (Alternatives Considered)

### 현행 유지 (이벤트별 1도큐먼트, `likedBy: Long`)
- 구현이 단순하고 쓰기 시 충돌이 없다.
- 동일 게시물 좋아요가 100개면 100개의 도큐먼트가 생성된다. 클라이언트가 집계 표현을 하려면 조회 후 직접 그룹핑해야 한다.
- **채택하지 않음**: 집계 표현을 위한 별도 연산 비용이 크고, 도큐먼트 수가 좋아요 수에 비례해 무한히 증가한다.

### 별도 컬렉션 분리 (`likes` 컬렉션 + 알림에는 카운트만 저장)
- 좋아요 데이터와 알림 데이터의 책임이 명확히 분리된다.
- 알림 렌더링 시 "누가 눌렀는지" 표시를 위해 `likes` 컬렉션을 추가 조회해야 한다. 단일 조회로 해결되지 않는다.
- **채택하지 않음**: 이 서비스의 범위는 알림 처리이며, 별도 좋아요 컬렉션을 관리하는 것은 책임 범위를 벗어난다.

### 알림에 카운트(`likeCount: int`)만 저장
- 도큐먼트 크기가 고정되어 배열 증가 문제가 없다.
- "A님 외 N명이 좋아요를 눌렀습니다" 표현에서 A님이 누구인지 알림 도큐먼트만으로 알 수 없다. 클라이언트가 최신 좋아요 유저를 표시하려면 별도 API가 필요하다.
- **채택하지 않음**: 알림만으로 렌더링이 완결되지 않아 조회 복잡도가 증가한다.

## 결과 (Consequences)

### 긍정적 결과
- 알림 도큐먼트 하나를 읽으면 "누가 좋아요를 눌렀는지"를 추가 조회 없이 바로 렌더링할 수 있다.
- 동일 게시물에 대한 좋아요 알림이 1개로 유지되므로 도큐먼트 수가 좋아요 수에 비례해 증가하지 않는다.
- 새 좋아요 발생 시 `occurredAt`과 `expiresAt`이 최신 시각으로 갱신되어 알림이 피드 상단으로 올라오는 동작이 자연스럽다.

### 부정적 결과 / 트레이드오프
- **동시성 위험**: 현재 구현은 `findLikeByPostId` → `addLikedId` → `save`의 read-modify-write 패턴이다. 두 명이 동시에 좋아요를 누르면 나중에 저장된 쪽이 앞서 추가된 ID를 덮어쓸 수 있다. MongoDB의 원자적 배열 업데이트(`$push`, `$addToSet`)를 사용하지 않는 현재 구조에서는 이 경쟁 조건이 잠재적으로 존재한다. 현재 서비스 규모(포트폴리오)에서는 허용하지만, 실 서비스 전환 시 반드시 해소해야 한다.
- **배열 무한 증가**: 좋아요가 매우 많은 게시물은 `likedIdsBy` 배열이 계속 커진다. MongoDB 도큐먼트의 최대 크기는 16MB이므로 이론적 한계가 존재한다. 좋아요 수가 수만 건에 이르는 규모라면 최근 N명만 보관하거나 별도 컬렉션으로 분리하는 방안을 재검토해야 한다.
- **기존 데이터 마이그레이션**: 이미 이벤트별로 생성된 `LikeNotification` 도큐먼트가 있다면, 게시물 단위로 병합하는 마이그레이션 스크립트가 필요하다.

# ADR-002: 이벤트 기반 아키텍처와 Kafka 도입

## 컨텍스트 (Context)

ADR-001에서 `notification-consumer`가 이벤트를 수신하는 구조로 결정되었다. 그런데 각 이벤트 생산자(댓글 서버, 리액션 서버, 팔로우 서버)와 `notification-consumer` 사이의 **통신 방식**을 결정해야 한다.

**REST API를 사용할 경우의 문제:**

*동기 REST:*
대형 인플루언서의 게시물에 초당 수십만 개의 댓글이 달리는 상황을 가정하면, 댓글 서버는 각 댓글 생성마다 `notification-consumer`에 동기적으로 HTTP 요청을 보내야 한다. `notification-consumer`가 처리량을 감당하지 못하고 죽으면 댓글 서버의 요청도 실패하며, 댓글 서버 자체의 응답 지연을 야기한다.

*비동기 REST (fire-and-forget):*
응답을 기다리지 않더라도, `notification-consumer`가 수많은 요청을 감당하지 못해 죽으면 여전히 문제가 발생한다. 더 나아가, `notification-consumer` 하나가 죽으면 댓글 서버뿐 아니라 리액션 서버, 팔로우 서버 모두에 영향을 주는 **단일 실패 지점(Single Point of Failure)**이 명확히 존재한다.

두 방식 모두 이벤트 생산자와 소비자 사이의 **강결합** 문제를 해결하지 못한다.

## 결정 (Decision)

서비스 간 통신을 **이벤트 기반 아키텍처(Event-Driven Architecture)**로 구성하고, 메시지 브로커로 **Apache Kafka**를 도입한다.

**동작 흐름:**
1. 댓글/리액션/팔로우 서버는 이벤트를 Kafka 토픽에 발행(produce)하고 이후에 더 이상 관심을 갖지 않는다.
2. `notification-consumer`는 독립적으로 Kafka에서 이벤트를 가져와(consume) 알림 데이터를 생성한다.

```yaml
# notification-consumer/application.yaml
spring.cloud.stream.kafka.binder.brokers: localhost:9092
spring.cloud.stream.bindings:
  comment-in-0.destination: comment
  like-in-0.destination: like
  follow-in-0.destination: follow
```

**RabbitMQ 대신 Kafka를 선택한 이유:**

Kafka는 **Append-Only 로그** 구조를 갖는다. 이벤트를 소비해도 삭제하지 않으며, 각 소비자는 자신이 마지막으로 읽은 오프셋(offset)을 관리한다.

- `notification-consumer`가 장애로 중단되더라도 이벤트는 Kafka에 보존된다.
- 재시작 시 중단된 오프셋부터 다시 소비하면 되므로 **이벤트 유실이 없다**.
- 장애 구간에서 이벤트가 실제로 유실되었는지 오프셋으로 확인할 수 있다.

RabbitMQ 등 전통적인 메시지 큐는 소비 시 메시지를 삭제하므로, 소비자 장애 시 처리되지 않은 메시지의 유실 위험이 있고 재처리를 위한 별도 전략이 필요하다.

## 대안 (Alternatives Considered)

**동기 REST API:**
- 생산자와 소비자의 강결합, 소비자 장애가 생산자에 직접 전파된다.
- 채택하지 않음.

**비동기 REST API (fire-and-forget):**
- 호출 시점의 결합은 낮아지지만, 소비자 과부하 및 단일 실패 지점 문제가 여전히 존재한다.
- 채택하지 않음.

**RabbitMQ:**
- 메시지 소비 후 삭제 → 내구성 부족, 재처리 어려움.
- Kafka의 Append-Only 특성이 이 프로젝트의 요구사항에 더 부합한다.
- 채택하지 않음.

## 결과 (Consequences)

### 긍정적 결과
- 생산자와 소비자 간 **느슨한 결합(Loose Coupling)**: 소비자 장애가 생산자에 영향을 주지 않는다.
- Append-Only 로그를 통한 **이벤트 내구성**: 소비자 재시작 시 offset 기반 자동 재처리가 가능하다.
- 각 서비스가 Kafka에 이벤트를 발행하는 것 외에는 알림 시스템에 대한 의존성이 없다.
- `max-attempts: 2` 설정으로 기본적인 재시도 전략이 내장되어 있다.

### 부정적 결과 / 트레이드오프
- Kafka + Zookeeper 인프라 운영 복잡도가 증가한다.
- 이벤트 스키마(`CommentEvent`, `LikeEvent`, `FollowEvent`) 변경 시 생산자와 소비자 간 호환성 관리가 필요하다.
- 이벤트 순서 보장이 필요한 경우 파티션 전략을 별도로 설계해야 한다.

### 후속 과제
- Dead Letter Queue(DLQ) 처리 전략 설계: `max-attempts` 초과 시 메시지 처리 방법
- 이벤트 스키마 버전 관리 전략 수립
- 파티션 키 전략 결정 (`userId` 기준 파티셔닝 등)
